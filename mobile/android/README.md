# Animus Link Android Host App (MVP)

This app is the Android Platform-0 host app with VPNService full-tunnel beta controls.

FFI linkage (UniFFI):
- Rust core API is defined by `crates/fabric-ffi/src/fabric.udl`.
- Kotlin bindings are generated by:
  - `cargo run --quiet -p fabric-ffi --bin generate-bindings -- kotlin mobile/android/app/src/main/kotlin`
- Rust shared libraries are produced into `app/src/main/jniLibs/<abi>/libfabric_ffi.so` by:
  - `mobile/android/scripts/build-rust-android.sh`
- Supported Android ABIs for MVP:
  - `arm64-v8a`
  - `x86_64`

Build (CI/MVP):
- Generate UniFFI Kotlin bindings.
- Build Rust shared libraries for required ABIs.
- Run `testDebugUnitTest assembleDebug`.

Relay/security notes:
- Full-tunnel uses `AnimusVpnService` + UniFFI tunnel controls in-process.
- Relay path requires signed relay tokens by default (no unsigned-token dev path for beta).
- Never log invite/token/key material in Android logs.
- Public beta policy is foreground-only and enforced as a foreground service notification.

Using VPN mode (manual device test):
1. Build/install debug app.
2. Enter:
  - `Relay addr` (`host:port`)
  - signed `relay token` (`animus://rtok/v1/...`)
  - gateway service (`gateway-exit` by default)
  - fail mode (`open_fast` default, optional `closed`)
  - DNS mode (`remote_best_effort` default)
3. Tap `Start Tunnel` and grant Android VPN permission.
4. Verify app shows `Tunnel: state=connected`.
5. Validate egress/DNS behavior with your operator test flow.
6. Tap `Stop Tunnel` to tear down.

Fail policy behavior:
- `open_fast` (default): if tunnel drops and does not recover quickly, VPN service stops within ~2s.
- `closed`: VPN stays up and traffic remains blocked until tunnel recovers or user stops it.

Permissions:
- `android.permission.INTERNET`
- `android.permission.FOREGROUND_SERVICE`
- system-managed `android.permission.BIND_VPN_SERVICE` for VPN service component

Play Store beta flow:
1. Download CI artifact `android-debug-apk` (or release `android-apk`).
2. Validate app launch + tunnel start/stop controls.
3. Validate relay-first full tunnel against signed-token relay backend.
4. Upload to Google Play Internal/Closed testing track.
5. Promote to wider track after telemetry + crash review.
